# Kempape — Authentication & Onboarding System Handoff

## Purpose

This document defines the invite-code authentication and onboarding system for the Kempape MVP.

The game is private and has only a small, admin-approved group of players.

The MVP does not need:

* Public registration.
* Email registration.
* Password creation.
* Password recovery.
* Social login.
* User-selected characters.

The admin creates every player before the festival and assigns each player's character.

Each player receives one private invite code that opens their assigned account.

---

# Core Authentication Rules

| Rule | Decision |
| ---- | -------- |
| Public registration | No |
| Email required | No |
| User-created password | No |
| Invite code per player | One active code |
| Player created by | Admin |
| Character assigned by | Admin |
| Persistent phone session | Yes |
| Code reusable for account recovery | Yes |
| Admin can revoke code | Yes |
| Admin can replace code | Yes |
| Anonymous spectators | No |

# Player Creation

Before the festival, the admin creates each player with:

```ts
type PlayerAccount = {
  id: string;
  displayName: string;
  characterId: string;

  totalXp: 0;
  currentLevel: 1;
  currentHp: 100;
  coins: 50;

  status: "active" | "disabled";
};
```

Starting inventory:

```text
Equipment: none
Consumables: none
Chaos Cards: none
```

The normal Daily Wheel spin is available when the festival begins.

---

# Invite Code Generation

Each player receives one randomly generated invite code.

Recommended format:

```text
KMP-7H9Q-X4MD-P8TR
```

Requirements:

* Generated with a cryptographically secure server-side random source.
* Long enough that guessing is impractical.
* Case-insensitive after normalization.
* Avoid ambiguous characters such as `0`, `O`, `1`, and `I`.
* Unique across all players.
* Never generated by the client.

Recommended normalized alphabet:

```text
ABCDEFGHJKLMNPQRSTUVWXYZ23456789
```

Recommended random section:

```text
16 characters
```

The `KMP-` prefix is cosmetic and may be removed before hashing.

---

# Secure Code Storage

Do not store the usable invite code as plain text in the database.

Recommended storage:

```ts
type PlayerInviteCode = {
  id: string;
  playerId: string;

  codeHash: string;
  codeLastFour: string;

  isActive: boolean;

  createdAt: string;
  revokedAt?: string;
  lastUsedAt?: string;
};
```

Recommended process:

1. Generate the random code.
2. Show it to the admin once.
3. Normalize it.
4. Hash it server-side.
5. Store only the hash and last four characters.
6. Give the original code privately to the player.

Because the invite code is a high-entropy random secret, a secure server-side SHA-256 or HMAC-SHA-256 lookup hash is appropriate.

The code must never be:

* Written to analytics.
* Printed in production logs.
* Returned by normal player APIs.
* Exposed to other players.

---

# Login Flow

The login screen contains:

```text
Invite code field
ENTER button
```

When the player submits a code:

1. Normalize the code.
2. Apply rate limiting.
3. Hash the normalized code on the server.
4. Find one active matching invite-code record.
5. Confirm the assigned player is active.
6. Create a secure player session.
7. Store the session in an `HttpOnly` secure cookie.
8. Update `lastUsedAt`.
9. Redirect to the correct game screen.

Do not return different errors for:

* Unknown code.
* Revoked code.
* Disabled player.

Use one generic message:

```text
Invalid or inactive invite code.
```

---

# Persistent Session

After successful login, the player should remain signed in on that phone.

Recommended cookie properties:

```text
HttpOnly
Secure
SameSite=Lax
Path=/
```

The cookie should contain an opaque random session token, not the invite code.

Recommended session record:

```ts
type PlayerSession = {
  id: string;
  playerId: string;

  tokenHash: string;
  sessionVersion: number;

  createdAt: string;
  lastSeenAt: string;
  expiresAt: string;

  revokedAt?: string;
};
```

The raw session token is sent only in the secure cookie.

The database stores only its hash.

The app restores the session automatically when reopened.

---

# Reusing the Invite Code

The invite code is reusable.

This allows a player to recover access when:

* Browser data was cleared.
* The app was reinstalled.
* The phone was changed.
* The session expired.

Submitting the same valid code opens the same pre-created player account.

It never creates a second player.

---

# Recommended Concurrent-Session Rule

For the MVP, use one active session family per player.

Recommended behavior:

```text
Logging in with the invite code on a new device
invalidates older sessions for that player.
```

This reduces accidental account sharing and prevents two phones from controlling one player simultaneously.

Multiple tabs using the same current browser session remain valid.

Implementation option:

```text
Increment player.sessionVersion
when a fresh invite-code login succeeds.
```

Any session containing an older version becomes invalid.

---

# Admin Revocation and Replacement

## Revoke Code

When the admin revokes a code:

* Set `isActive = false`.
* Store `revokedAt`.
* Reject future logins with that code.
* Existing sessions may also be revoked.

## Replace Code

When the admin replaces a code:

1. Revoke the previous code.
2. Generate a new secure random code.
3. Store the new hash.
4. Show the new code to the admin once.
5. Invalidate old sessions when appropriate.

The player's game progress is unchanged.

Only the login credential changes.

---

# Player Disable

The admin may disable a player account.

When disabled:

* Invite-code login fails.
* Existing sessions stop working.
* The player's data remains stored.
* The player remains recoverable by the admin.

Disabling is different from deleting.

Do not permanently delete player data during the live festival.

---

# Festival Access States

Authentication may remain available before and after the active game window.

## Before Festival Start

A valid player may log in and see:

```text
Festival starts in:
[countdown]
```

Gameplay mutations are disabled.

## During Festival

All normal game rules are active.

## After Festival End

A valid player may log in and see:

* Final ranking.
* Final character state.
* Read-only history where implemented.

Gameplay mutations are disabled.

The server, not the client, must enforce the active festival window.

# Authorization Rules

Authentication answers:

```text
Who is this player?
```

Authorization answers:

```text
What is this player allowed to read or change?
```

Recommended access:

## Player May Read

* Their complete player state.
* Their inventory.
* Their cooldowns.
* Their quest progress.
* Their chest and wheel results.
* The shared Action Pool.
* Public ranking fields for all players.

## Player May Not Read

* Another player's inventory.
* Another player's invite-code data.
* Another player's private session data.
* Admin-only logs.
* Hidden moderation information.

## Player May Change

Only their own state through approved server commands.

The client must not directly update:

* XP.
* HP.
* Coins.
* Inventory quantities.
* Equipment supply.
* Chest results.
* Wheel results.
* Hospital state.

Those changes occur through server-side transactions.

---

# Rate Limiting

Invite-code attempts must be rate-limited.

Recommended MVP starting rule:

```text
Maximum 5 failed attempts
per device or network address
within 10 minutes.
```

After the limit:

```text
Too many attempts. Try again later.
```

Successful sessions should not repeatedly submit the invite code.

---

# Audit Events

Record security-relevant events without recording the secret code.

Examples:

```text
invite_login_succeeded
invite_login_failed
invite_code_revoked
invite_code_replaced
session_revoked
player_disabled
```

Recommended audit fields:

```ts
type AuthenticationAuditEvent = {
  id: string;
  eventType: string;
  playerId?: string;
  inviteCodeId?: string;
  createdAt: string;
  metadata?: Record<string, unknown>;
};
```

Never store the submitted raw invite code in audit metadata.

# Idempotency and Security Rules

| Rule | Purpose |
| ---- | ------- |
| Codes generated server-side | Prevent predictable codes |
| Plain code not stored | Reduce credential exposure |
| Secure cookie session | Prevent normal JavaScript access |
| Server validates every command | Prevent client manipulation |
| Generic login error | Avoid revealing valid accounts |
| Rate-limited attempts | Reduce guessing |
| One code maps to one player | Prevent duplicate accounts |
| Reusing a code restores the same account | Support recovery |
| Revocation supported | Recover from a leaked code |
| New login invalidates older sessions | Reduce account sharing |
| Admin actions audited | Support debugging and recovery |
| No invite codes in logs | Protect credentials |

# Acceptance Tests

## Correct Code

```text
Given an active player and valid invite code,
when the code is submitted,
then a secure session is created
and the assigned player account opens.
```

## Incorrect Code

```text
Given an unknown code,
when it is submitted,
then no session is created
and a generic error is shown.
```

## Reused Code

```text
Given a player previously logged in,
when the same valid code is used again,
then the same player account opens
and no new player is created.
```

## Persistent Session

```text
Given a valid session cookie,
when the app is closed and reopened,
then the player is restored
without entering the invite code again.
```

## New Device Login

```text
Given a valid player session on Device A,
when the invite code is used on Device B,
then Device B becomes authenticated
and the older session is invalidated.
```

## Revoked Code

```text
Given an invite code has been revoked,
when it is submitted,
then login fails.
```

## Replacement Code

```text
Given a replacement code is generated,
then the old code no longer works,
the new code opens the same player,
and the player's progress is unchanged.
```

## Disabled Player

```text
Given a player is disabled,
then their code and existing sessions
cannot access the game.
```

## Before Start

```text
Given a valid player logs in before festivalStartAt,
then the player sees the countdown
and cannot mutate game state.
```

## After End

```text
Given a valid player logs in after festivalEndAt,
then the app is read-only
and no gameplay command succeeds.
```

# Final MVP Rule

```text
The admin pre-creates every player and assigns a character.

Each player receives one secure reusable invite code.

The code logs the player into that exact account.

A secure phone session is created and restored automatically.

There is no public registration, email, or user-created password.

The admin can revoke or replace a code without deleting progress.
```
